## Three Pillars

The engine is built around three pillars:

- **Rendering** — Canvas2D (2D), Canvas3D (3D), frame capture, and the full PBR pipeline
- **AI Interface** — MCP server with 17 tools (visual / control / state / scene) enabling agents to see and act
- **Runtime** — `GameCallbacks` lifecycle + `bevy_app` schedules for ECS systems

## Crate Structure

```
Engine (umbrella): anvilkit
Runtime:           anvilkit-app (GameCallbacks, schedules, state, plugins)
Rendering:         anvilkit-render (Canvas2D, Canvas3D, PBR pipeline, capture, transform)
Core:              anvilkit-core (math, time, error, persistence)
Assets:            anvilkit-assets
Input:             anvilkit-input
Audio:             anvilkit-audio
AI Describe:       anvilkit-describe + anvilkit-describe-derive
AI MCP:            anvilkit-mcp (17 tools: spawn_object, press_key, capture, save_scene, etc.)

Game-level (not in umbrella):
  anvilkit-camera, anvilkit-gameplay, anvilkit-data
```

> **Note:** `anvilkit-ecs` no longer exists. Its schedules, plugin system, and state machine have moved into `anvilkit-app`.

## ECS Schedule Phases

| Phase | Timing | Typical Systems |
|-------|--------|-----------------|
| `Startup` | Once at init | Resource setup, initial spawn |
| `Main` | Every frame, first | (reserved for game-wide logic) |
| `PreUpdate` | Every frame, before Update | DeltaTime sync, parent-child hierarchy sync |
| `FixedUpdate` | 0-N per frame (accumulator) | Deterministic physics, fixed-rate gameplay |
| `Update` | Every frame, main logic | Input, physics, game logic, camera |
| `PostUpdate` | Every frame, after Update | Transform propagation, render extract |
| `Cleanup` | Every frame, last | (reserved) |

## Module Overview

### anvilkit-core

Foundation type library: math utilities (Vec3, Mat4, Transform, GlobalTransform, interpolation), time system (Time, Timer), error types (AnvilKitError with category-preserving `with_context`), persistence system (SaveManager, Settings, WorldStorage — all returning `Result<_, AnvilKitError>`). Aabb axis-aligned bounding box, Frustum, and raycast utilities (screen_to_ray, ray_plane_intersection, ray_sphere_intersection) — moved from render crate in v0.3. Persistence error category (AnvilKitError::Persistence) with category-preserving with_context.

### anvilkit-render

Unified 2D/3D rendering with agent-friendly entry points:
- **Canvas2D**: Immediate-mode 2D drawing (rects, text, sprites)
- **Canvas3D**: Scene-graph 3D API with PBR materials and camera control
- **Frame Capture**: `capture_frame()` returns rendered pixels — lets AI agents see the game
- PBR material system (Cook-Torrance BRDF, 5 texture maps)
- HDR rendering (Rgba16Float) + ACES Filmic tone mapping
- Bloom (13-tap downsample + 9-tap tent upsample, 5 mip levels)
- SSAO (hemisphere sampling, half-res, depth-reconstructed normals)
- Cascade Shadow Maps (3 cascades, D2Array, PCF)
- IBL (BRDF LUT + environment irradiance)
- MSAA 4× anti-aliasing
- GPU skeletal animation (Linear Blend Skinning, 128 joints)
- Frustum culling + GPU instancing + batched draw sorting
- Sprite, Line, Particle, Text, UI renderers
- StandardMaterial component (PBR with builder pattern)
- PostProcessSettings (SSAO/DOF/MotionBlur/Bloom/ColorGrading)
- Orthographic/Perspective projection enum, multi-camera priority
- PipelineCache/BindGroupCache
- Point/spot shadow system (ShadowAtlas)
- events.rs split into 6 files (events/ directory), draw.rs split into 5 files (draw/ directory)
- CachedBuffer shared utility replacing 5 duplicate cached_vb patterns
- ProjectionUniform shared type replacing 5 duplicate ortho structs
- DebugRenderer merged with LineRenderer

### anvilkit-assets

Asset pipeline: glTF model loading (mesh, material, skeleton, animation), AssetServer (sync + async loading, deduplication), AssetHandle (Arc refcount), procedural mesh generation, file watching hot reload. DependencyGraph for asset dependency tracking (AssetId-based). AssetCache integration (content hash deduplication). Cascade unload through dependency graph. insert_parsed/get_parsed for typed asset storage.

### anvilkit-audio

Audio system based on `rodio`: AudioSource component with ECS integration, AudioBus (master/music/sfx/voice volume mixing), spatial audio (distance attenuation via AudioListener), looping and pitch control. NonSend resource pattern for thread-safe macOS CoreAudio. Stereo panning via listener right vector. Asset integration via AudioSource.asset_id.

### anvilkit-input

Input abstraction: InputState (keyboard/mouse pressed/just_pressed/released, mouse_delta, scroll_delta), GamepadState (16 buttons, 6 axes), ActionMap (logical action → physical binding), ActionId (zero-allocation lookups), AxisBinding (keyboard/gamepad analog axes). InputPlugin for automatic resource initialization. ActionMap with apply_overrides for key rebinding. Gamepad event forwarding.

### anvilkit-camera

Production camera system with 5 modes (FirstPerson, ThirdPerson, Orbit, Free, Rail). Features: OrbitState component, CameraRig entity follow, SpringArm collision avoidance, trauma-based Perlin noise shake, CameraTransition blending (5 easing curves), LookAtTarget soft constraint, CameraRail dolly (Catmull-Rom), InputCurve dead zone + response curves. 8-system pipeline, frame-rate independent smoothing.

### anvilkit-app

Game application runner that eliminates boilerplate. `AnvilKitApp::run()` owns the winit event loop, input forwarding, DeltaTime management, and frame lifecycle. Games implement `GameCallbacks` (init, post_update, render, on_resize, on_window_event) and pass a `GameConfig` (title, size, vsync). Provides `GameContext` for access to the ECS `App` and `RenderApp`. Since the refactor, anvilkit-app also contains schedules, ECS plugin system, and state machine (previously in anvilkit-ecs, which no longer exists).

### anvilkit-describe

AI-first introspection: the `Describe` trait and its derive macro allow engine types to emit structured descriptions consumable by AI agents. The companion `anvilkit-describe-derive` crate provides the proc macro.

### anvilkit-mcp

MCP (Model Context Protocol) server with 17 tools that exposes engine state and controls to AI agents. Tool categories: **visual** (capture_frame, describe_scene), **control** (press_key, move_mouse, spawn_object), **state** (get_component, set_component), **scene** (save_scene, load_scene). Enables agent-driven game development workflows where the agent can see and interact with the running game.

### anvilkit-gameplay

Optional, feature-gated gameplay building blocks: `stats` (generic stat system with additive/multiplicative/override modifiers), `health` (Health component with regen, DamageEvent, HealEvent, DeathEvent, health_system), `inventory` (slot-based and stackable items), `cooldown` (ability/action cooldown timers), `status-effect` (duration-based effects with stacking policies), `entity-pool` (object pool for entity recycling).

### anvilkit-data

Data-driven configuration and localization: `DataTable` (typed key-value table loaded from RON or JSON), `Locale` (translation lookup with fallback chain), `DataTablePlugin` for ECS integration.

## Coordinate System

AnvilKit uses a **left-handed coordinate system**:

- +X points right
- +Y points up
- +Z points into the screen (forward)

Projection uses `perspective_lh` and `look_at_lh`.
